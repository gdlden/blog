# Phase 6.1: Fix Blog Create Bug - Research

**Researched:** 2026-04-14
**Domain:** Kratos/Go dependency injection (Wire), Vue 3 frontend error handling
**Confidence:** HIGH

## Summary

The "创建失败" bug is a gap-closure issue from Phase 6. Investigation confirms three contributing factors: (1) `wire_gen.go` is outdated because it was never regenerated after `UpdatePost`/`DeletePost` were added to `PostUsecase`, but the current file on disk actually appears syntactically valid because `wire` was run at some point after the service layer was added; (2) `go test ./cmd/blog/...` fails due to a `go vet` error in `main.go` (`non-constant format string in fmt.Printf`), which breaks the all-packages test command even though `go build` succeeds; (3) the frontend response interceptor expects `res.data.code === "200"`, but Kratos HTTP replies for this service do not wrap responses with a `code` field, so successful responses are incorrectly rejected as failures and the generic toast "创建失败" is shown.

**Primary recommendation:** Fix `main.go` format string so `go test ./...` passes, regenerate `wire_gen.go` to ensure it matches the current provider graph, and update the frontend Axios interceptor to stop checking for a non-existent `code` field (or make the check defensive).

## Architecture Patterns

### Wire Regeneration Pattern (Kratos)
**What:** After any change to constructors, provider sets, or new usecase/service methods that affect the dependency graph, run `wire` in `cmd/blog/` to regenerate `wire_gen.go`.
**When to use:** Whenever `biz`, `service`, `data`, or `server` packages gain/lose constructors or change signatures.
**Example:**
```bash
cd blog/cmd/blog
wire
# or: go run github.com/google/wire/cmd/wire@latest
```

### Frontend Response Interceptor Anti-Pattern
**What:** The current interceptor assumes all successful responses contain `res.data.code === "200"`.
**Why it's wrong:** Kratos HTTP handlers for this project return the protobuf message directly as JSON (e.g., `{"id":"","title":"","content":""}`) without a wrapping `code`/`message` envelope. This causes the interceptor to reject successful 200 responses.
**What to do instead:** Either remove the `code` check entirely (rely on HTTP status), or make it defensive:
```typescript
if (res.data && res.data.code && res.data.code !== "200") {
  return Promise.reject(res.data);
}
return res.data;
```

## Common Pitfalls

### Pitfall 1: `go test` fails while `go build` passes
**What goes wrong:** `go test` runs `go vet` by default. `main.go:79` uses `fmt.Printf("data_database_source:" + value)` which triggers `non-constant format string` vet error. This causes `go test ./...` to report a build failure for `cmd/blog`, even though `go build ./cmd/blog` succeeds.
**Why it happens:** `fmt.Printf` with a concatenated string is flagged as a potential format-string vulnerability.
**How to avoid:** Use `fmt.Print("data_database_source:" + value)` or `fmt.Printf("data_database_source:%s", value)`.
**Warning signs:** `go test ./...` reports "build failed" for a package that compiles fine with `go build`.

### Pitfall 2: wire_gen.go is gitignored but stale
**What goes wrong:** `.gitignore` ignores `blog/cmd/blog/wire_gen.go`. A fresh clone won't have it, and an existing working copy may hold a stale version that looks correct but doesn't match the latest code.
**Why it happens:** Generated files are excluded from version control per project convention.
**How to avoid:** Always run `wire` after pulling or after changing DI-relevant code. Verify with `wire check`.
**Warning signs:** `undefined: wireApp` on a fresh machine, or subtle runtime panics from missing service registrations.

### Pitfall 3: Frontend interceptor rejects valid responses
**What goes wrong:** `request.ts` checks `res.data.code === "200"`. For endpoints that return plain protobuf JSON without a wrapper, `res.data.code` is `undefined`, the condition is false, and the interceptor rejects the response.
**Why it happens:** Some legacy/project APIs may use a unified response wrapper with `code`, but Kratos-generated HTTP bindings return the message shape directly.
**How to avoid:** Make the interceptor defensive: only reject if `code` exists and is not `"200"`.
**Warning signs:** Network tab shows HTTP 200 with correct JSON, but the UI shows an error toast.

### Defensive response interceptor
```typescript
// src/utils/request.ts
instance.interceptors.response.use(
  res => {
    if (res.data && res.data.code && res.data.code !== "200") {
      return Promise.reject(res.data);
    }
    return res.data;
  },
  err => {
    if (err.response) {
      console.log(err.response.status, err.response.data);
      alert(err.response.data?.message || "请求失败");
    }
    return Promise.reject(err);
  }
);
```
